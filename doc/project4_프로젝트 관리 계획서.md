## 1. 개요 (Introduction)

### 1.1 문서의 목적

본 문서는 DevTrack 개발 프로젝트의 전반적인 관리 계획을 정의하고, 프로젝트 수행 과정에서 팀원 간 공통된 기준과 지침을 제공하기 위해 작성되었다. 개발 일정, 팀 구성, 품질 관리, 리스크 관리 등 프로젝트 관리에 필요한 모든 사항을 포함한다.

본 문서는 다음 범위에 적용된다:
- 프로젝트 개요 및 핵심 기능 정의
- 개발 절차 및 일정 계획
- 팀 구성 및 역할 분담
- 품질 관리 및 변경 관리 방법
- 산출물 정의 및 관리 방법

### 1.2 프로젝트 개요

**시스템 명칭:** DevTrack — 소프트웨어 개발 프로젝트 통합 관리 시스템

**시스템 설명:**  
소프트웨어 개발 조직에서 요구사항 관리, 일정 추적, 산출물 버전 관리, 팀원 역할 및 권한 관리를 하나의 플랫폼에서 통합 제공하는 시스템이다. 여러 도구에 분산된 개발 관리 작업을 단일 플랫폼으로 통합하여 투명한 협업 환경을 구축하는 것을 목표로 한다.

**핵심 기능:**

| ID | 기능명 | 설명 |
|----|--------|------|
| F01 | 요구사항 등록 및 추적 | 고객 요구사항 등록 및 변경 이력 추적 |
| F02 | 일정 및 마일스톤 관리 | 프로젝트 일정 등록 및 진행 상황 시각화 |
| F03 | 팀원 역할 및 권한 관리 | 팀원별 역할 지정 및 접근 권한 설정 |
| F04 | 산출물 버전 관리 | 문서·설계서 수정 이력 보관 및 버전 복원 |
| F05 | 진행 상황 대시보드 | 전체 진행률·완료 태스크·잔여 일정 통합 조회 |
| F06 | 알림 및 공지 기능 | 주요 변경 사항 발생 시 자동 알림 제공 |

## 2. 개발 계획 (Development Plan)

### 2.1 프로세스 모델 및 개발 방법론

**프로세스 모델: Agile — Scrum**

DevTrack은 초기 요구사항이 완전히 확정되지 않은 상태에서 시작되며, 개발 과정에서 지속적인 기능 추가 및 변경이 예상된다. 이에 따라 짧은 스프린트 단위로 기능을 개발하고 팀원 간 피드백을 빠르게 반영할 수 있는 Scrum 모델을 채택한다.

**설계 방법론: 객체지향 방법론**

요구사항 분석 및 설계 단계에서는 객체지향 방법론을 적용한다. 클래스 단위의 모듈화를 통해 분석, 테스트, 유지보수, 확장에 유리한 구조를 추구한다.

### 2.2 작업 분할도 (WBS)

```

├── 1. 요구사항 관리

│   └── 요구사항 분석서 작성
├── 2. 설계

│   ├── DB 스키마 설계

├── 3. 코딩
│   ├── 요구사항 등록·추적 모듈 구현
│   ├── 일정·마일스톤 관리 모듈 구현
│   ├── 권한 관리 모듈 구현
│   ├── 산출물 버전 관리 모듈 구현

│   ├── 단위 테스트
```

## 3. 조직 구성 (Organization)

### 3.1 팀 구조

**분산형(민주주의적) 팀 구조**를 채택한다. 팀장의 역할은 존재하지만, 모든 팀원이 동등한 발언권을 가지며 주요 의사결정은 팀원 합의로 진행한다. 요구사항과 기능이 다양하고 복잡한 DevTrack의 특성을 고려하여 각 팀원이 전문 영역을 담당하는 수평적 협업 구조를 지향한다.

### 3.2 책임 및 역할

| 역할 | 주요 책임 |
|------|-----------|
| 팀장 | 프로젝트 전반 진행 조율, 문서 관리, 의견 충돌 시 최종 결정 |
| 팀원 (BE) | 백엔드 API 설계 및 구현, DB 관리 |
| 팀원 (FE) | UI/UX 설계 및 프론트엔드 구현 |
| 팀원 (QA) | 테스트 계획 수립 및 품질 검토 |
| 팀원 공통 | 요구사항 분석 참여, 코드 리뷰, 문서 작성 |

## 4. 기술 관리 계획 (Technical Management)

### 4.1 버전 관리

- **소프트웨어:** 파일명 맨 앞에 버전을 기재하고 GitHub를 통해 관리한다.  
  예) `v1_devtrack_auth_module.py`
- **문서:** 파일명 맨 뒤에 날짜를 기재하고, 문서 내 제·개정 이력을 함께 작성한다.  
  예) `[dayeoniiiki]DevTrack_관리계획서_260425_Doc-001.md`
- **기타 파일:** 파일명 앞에 날짜를 기재한다.  
  예) `260425_erd_diagram.png`

### 4.2 형상 관리

- 산출물 명명 규칙: `[팀명]시스템명_문서명_YYMMDD_Doc-버전`
- 소프트웨어 코드: GitHub 저장소(`dayeoniiiki/SE`)에서 브랜치 전략으로 관리
  - `main`: 배포 가능한 안정 버전
  - `develop`: 개발 통합 브랜치
  - `feature/기능명`: 기능별 작업 브랜치
- 문서: GitHub `doc/` 디렉토리 및 팀원 공유 클라우드에 저장

### 4.3 기술 관리

- 기능 변경 시 팀원 전체의 합의(다수결)가 필요하다.
- 변경 사항 발생 시 변경 전·후 내용을 반드시 GitHub Issue 또는 PR로 기록한다.
- 결정된 변경 사항은 담당 교수에게 보고한다.
- 의견 충돌로 진행이 멈출 경우 팀장이 최종 결정권을 가진다.

## 5. 품질 관리 계획 (Quality Control)

### 5.1 품질 목표

| 품질 요소 | 목표 |
|-----------|------|
| 기능성 | 정의된 6개 핵심 기능 100% 구현 |
| 신뢰성 | 주요 기능 오류율 5% 미만 |
| 사용성 | 신규 사용자가 주요 기능을 10분 이내 습득 가능 |
| 유지보수성 | 모듈별 독립성 확보, 코드 주석 작성률 80% 이상 |

### 5.2 검토 방법 및 주기

| 검토 유형 | 주기 | 참여자 |
|-----------|------|--------|
| 코드 리뷰 (PR) | 기능 구현 완료 시마다 | 팀원 전체 |
| 문서 리뷰 | 산출물 작성 완료 시 | 팀원 전체 |
| Sprint Review | 스프린트 종료 시 | 팀원 전체 |
| 전체 품질 점검 | 통합 테스트 단계 | 팀원 전체 |

### 5.3 품질 관리 적용 기법

- **인스펙션:** 주요 산출물(요구사항 정의서, 설계서 등) 작성 완료 후 팀원 전체가 체크리스트 기반으로 오류 검출
- **코드 리뷰:** GitHub Pull Request를 통해 기능 병합 전 최소 1인 이상의 승인 필수
- **테스트:** 단위 테스트 → 통합 테스트 → 시스템 테스트 순으로 진행

### 5.4 리스크 관리

| No | 리스크 | 심각도 | 발생 가능성 | 완화 계획 | 대안 |
|----|--------|--------|-------------|-----------|------|
| 1 | 팀원 이탈 | High | Low | 작업 문서화 철저, 지식 공유 | 남은 팀원 간 작업 재분배 |
| 2 | 요구사항 변경 | Mid | High | 스프린트 단위 유연한 계획 수립 | 베이스라인 기준 변경 범위 통제 |
| 3 | 일정 지연 | Mid | Mid | 주간 진척도 점검, 버퍼 기간 확보 | 우선순위 낮은 기능 다음 스프린트로 이월 |
| 4 | 기술 미숙 | Low | Mid | 스프린트 전 사전 학습 | 팀원 간 페어 프로그래밍 |

## 6. 개발 환경 (Development Environment)

### 6.1 요구되는 소프트웨어 스펙

| 구분 | 항목 | 버전/사양 |
|------|------|-----------|
| OS | Windows / macOS / Linux | 최신 안정 버전 |
| 언어 | Python / JavaScript | Python 3.10+, Node.js 18+ |
| 프레임워크 | Django / React | Django 4.x, React 18.x |
| DB | PostgreSQL | 14.x 이상 |
| 형상 관리 | Git + GitHub | 최신 버전 |
| 협업 도구 | Jira (confluence) | 웹 기반 |

### 6.2 요구되는 하드웨어 스펙

| 구분 | 사양 |
|------|------|
| CPU | Intel i5 이상 (8세대 이상 권장) |
| RAM | 8GB 이상 |
| 저장장치 | SSD 256GB 이상 |
| 네트워크 | 인터넷 접속 가능 환경 |

### 6.3 개발 공간 및 보안 요구사항

- 개발은 개인 노트북 및 팀원 공유 원격 저장소(GitHub)를 기반으로 진행한다.
- 소스 코드는 Private 또는 Public 저장소로 GitHub에서 관리한다.
- 민감 정보(API 키, 비밀번호 등)는 `.env` 파일로 분리하여 저장소에 커밋하지 않는다.
