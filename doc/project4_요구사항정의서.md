## 1. 서론

### 1.1 문서의 목적 및 범위

본 문서는 DevTrack 소프트웨어 개발 프로젝트 통합 관리 시스템의 요구사항을 체계적으로 정의하기 위해 작성되었다. 수집된 요구사항을 기능적 요구사항, 비기능적 요구사항, 인터페이스 요구사항으로 분류하여 기록하며, 이후 요구사항 분석서 및 설계서 작성의 기준 문서로 활용된다.

본 문서는 다음 범위에 적용된다.

- DevTrack 시스템의 기능적·비기능적·인터페이스 요구사항 정의
- 요구사항 추적성 매트릭스를 통한 요구사항 관리
- 개발 및 품질 관리 기준 제공

### 1.2 대상 시스템 개요

DevTrack은 소프트웨어 개발 조직에서 요구사항 관리, 일정 추적, 산출물 버전 관리, 팀원 역할 및 권한 관리를 하나의 플랫폼에서 통합 제공하는 시스템이다. 여러 도구에 분산된 개발 관리 작업을 단일 플랫폼으로 통합하여 투명한 협업 환경 구축을 목표로 한다.

| ID | 기능명 | 설명 |
|----|--------|------|
| F01 | 요구사항 등록 및 추적 | 고객 요구사항 등록 및 변경 이력 추적 |
| F02 | 일정 및 마일스톤 관리 | 프로젝트 일정 등록 및 진행 상황 시각화 |
| F03 | 팀원 역할 및 권한 관리 | 팀원별 역할 지정 및 접근 권한 설정 |
| F04 | 산출물 버전 관리 | 문서·설계서 수정 이력 보관 및 버전 복원 |
| F05 | 진행 상황 대시보드 | 전체 진행률·완료 태스크·잔여 일정 통합 조회 |
| F06 | 알림 및 공지 기능 | 주요 변경 사항 발생 시 자동 알림 제공 |

### 1.3 용어 정의

| 용어 | 정의 |
|------|------|
| 기능적 요구사항 | 사용자의 업무 처리와 직접 관련되어 시스템이 반드시 수행해야 하는 기능 요구사항 |
| 비기능적 요구사항 | 시스템의 성능, 보안, 운영 환경 등 품질 속성에 관한 요구사항 |
| 인터페이스 요구사항 | 시스템과 외부 시스템 또는 사용자 인터페이스 간의 연동 요구사항 |
| RBAC | Role-Based Access Control — 역할 기반 접근 제어 |
| 마일스톤 | 프로젝트 내 주요 완료 시점 또는 산출물 확정 시점 |
| 베이스라인 | 관련자 합의를 거쳐 확정된 산출물로, 이후 작업의 기준점 |
| RDD | Requirements Definition Document — 요구사항 정의서 (본 문서) |
| SRS | Software Requirements Specification — 요구사항 분석서 |

## 2. 요구사항

### 2.1 기능적 요구사항

| ID | 그룹 | 요구사항 내용 | 우선순위 |
|----|------|--------------|---------|
| FR-001 | 요구사항 관리 | 사용자는 요구사항 제목, 설명, 유형, 우선순위를 입력하여 요구사항을 등록한다. | 상 |
| FR-002 | 요구사항 관리 | 사용자는 등록된 요구사항의 상태를 미확인/확인/구현중/완료로 변경한다. | 상 |
| FR-003 | 요구사항 관리 | 시스템은 요구사항의 변경 이력(변경자, 변경일, 변경 내용)을 자동으로 기록한다. | 상 |
| FR-004 | 요구사항 관리 | 사용자는 요구사항을 키워드, 유형, 상태로 검색 및 필터링한다. | 중 |
| FR-005 | 일정 관리 | 사용자는 프로젝트 일정(시작일, 종료일, 마일스톤)을 등록한다. | 상 |
| FR-006 | 일정 관리 | 시스템은 등록된 일정을 간트 차트 형식으로 시각화하여 제공한다. | 중 |
| FR-007 | 일정 관리 | 사용자는 마일스톤 달성 여부를 완료/미완료로 표시한다. | 상 |
| FR-008 | 팀원 관리 | 관리자는 팀원의 역할(팀장/개발자/QA/뷰어)을 지정한다. | 상 |
| FR-009 | 팀원 관리 | 시스템은 역할에 따라 기능 접근 권한을 차등 적용한다. | 상 |
| FR-010 | 팀원 관리 | 관리자는 팀원을 프로젝트에 초대하거나 제외한다. | 중 |
| FR-011 | 산출물 관리 | 사용자는 산출물 파일을 업로드하고 버전 번호와 설명을 함께 등록한다. | 상 |
| FR-012 | 산출물 관리 | 사용자는 이전 버전의 산출물을 조회하고 다운로드한다. | 상 |
| FR-013 | 산출물 관리 | 사용자는 특정 버전의 산출물로 복원 요청을 한다. | 중 |
| FR-014 | 대시보드 | 시스템은 전체 진행률, 완료된 태스크 수, 잔여 일정을 대시보드에 표시한다. | 상 |
| FR-015 | 대시보드 | 사용자는 대시보드에서 자신에게 할당된 태스크 목록을 확인한다. | 상 |
| FR-016 | 알림 | 시스템은 요구사항 상태 변경, 마일스톤 도래, 산출물 업로드 시 관련 팀원에게 알림을 발송한다. | 상 |
| FR-017 | 알림 | 사용자는 알림 수신 항목을 개인별로 설정한다. | 하 |

### 2.2 비기능적 요구사항

| ID | 분류 | 요구사항 내용 | 우선순위 |
|----|------|--------------|---------|
| NFR-001 | 운영 환경 | 시스템은 Windows, macOS, Linux 운영체제 환경에서 동작한다. | 상 |
| NFR-002 | 운영 환경 | 시스템은 Chrome, Firefox, Edge 최신 버전의 웹 브라우저에서 동작한다. | 상 |
| NFR-003 | 성능 | 시스템은 모든 사용자 요청에 대해 3초 이내에 응답한다. | 상 |
| NFR-004 | 성능 | 시스템은 동시 접속자 50명 이상의 환경에서도 정상적으로 동작한다. | 중 |
| NFR-005 | 신뢰성 | 시스템의 주요 기능 오류율은 5% 미만으로 유지된다. | 상 |
| NFR-006 | 보안 | 시스템은 모든 사용자 비밀번호를 암호화하여 저장한다. | 상 |
| NFR-007 | 보안 | 시스템은 역할 기반 접근 제어(RBAC)를 적용하여 권한 외 기능 접근을 차단한다. | 상 |
| NFR-008 | 보안 | 민감 정보(API 키, 비밀번호)는 환경 변수(.env)로 분리하여 저장소에 포함하지 않는다. | 상 |
| NFR-009 | 사용성 | 신규 사용자는 별도의 교육 없이 10분 이내에 주요 기능(요구사항 등록, 일정 조회)을 사용할 수 있어야 한다. | 중 |
| NFR-010 | 유지보수성 | 시스템은 모듈별 독립성을 확보하여 단일 모듈의 변경이 타 모듈에 영향을 미치지 않도록 한다. | 중 |
| NFR-011 | 문화·정책 | 시스템은 한국어 및 영어 인터페이스를 지원한다. | 중 |

### 2.3 인터페이스 요구사항

## 5. 기타 고려 사항

- 모든 요구사항은 단문·5형식으로 작성하며, 모호한 표현(예: '빠르게', '충분히')은 사용하지 않는다.
- 요구사항 변경이 발생할 경우 팀원 합의 후 본 문서를 갱신하고 제·개정 이력을 기록한다.
- TBD(To Be Determined) 항목이 남아있는 경우 해당 Sprint 시작 전까지 확정한다.
- 본 문서는 스프린트 진행 중 요구사항 분석서(SRS) 작성 시 기준 문서로 사용된다.
